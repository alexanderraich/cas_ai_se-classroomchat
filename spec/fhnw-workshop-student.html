Hooks, Pre-commit & TDD

2:30 – 3:30 · 60 minutes

Hooks are the only way to get 100% enforcement

Hooks vs Prompts

DimensionHookPrompt (CLAUDE.md)
Guarantee100% — runs every time~90% — probabilistic
Best forSecurity, policy, gatesStyle, conventions
Blockingexit 2 blocks the tool callCannot block
FeedbackJSON on stdout to ClaudeRead at session start
CRITICAL: exit 1 does NOT block — it is treated as a hook error.
exit 2 is the ONLY way to deny a tool call.

Hook: block-secrets.sh

bash 
#!/bin/bash
# PostToolUse hook — block secret leaks in file writes
INPUT=$(cat </dev/stdin)
CONTENT=$(echo "$INPUT" | jq -r '.tool_input.content // .tool_input.command // ""')

# Check for AWS keys or OpenAI-style API keys
if echo "$CONTENT" | grep -qiE '(AKIA[0-9A-Z]{16}|sk-[a-zA-Z0-9]{40,})'; then
  echo '{"decision":"block","reason":"Secret pattern detected in output"}'
  exit 2
fi

echo '{"decision":"allow"}'
exit 0

Hook: quality-gate.sh

bash 
#!/bin/bash
# PreToolUse hook — require tests pass before any commit
INPUT=$(cat </dev/stdin)
TOOL=$(echo "$INPUT" | jq -r '.tool_name')

if [ "$TOOL" = "Bash" ]; then
  CMD=$(echo "$INPUT" | jq -r '.tool_input.command')
  if echo "$CMD" | grep -q "git commit"; then
    # Run tests and check for pass
    if ! npm test 2>&1 | tail -1 | grep -q "passed"; then
      echo '{"decision":"block","reason":"Tests must pass before commit"}'
      exit 2
    fi
  fi
fi

echo '{"decision":"allow"}'
exit 0

Hook Wiring: settings.json

json 
{
  "hooks": {
    "PostToolUse": [

        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/block-secrets.sh"
          }
        ]
      }

    "PreToolUse": [
      {
        "matcher": { "tool_name": "Bash" },
        "hooks": [
          {
            "type": "command",

          }

      }
    ]
  }
}

TDD Flow with Claude

Red — Write a failing test that describes the desired behavior
Green — Ask Claude to make the test pass (and only that test)
Refactor — Review the implementation and clean up
Next — Repeat for the next behavior
TDD with Claude works best when you write the test and let Claude write the implementation. You define WHAT, Claude figures out HOW.
prompt 
I wrote this failing test in src/utils.test.ts:

  test('slugify converts spaces to hyphens', () => {
    expect(slugify('hello world')).toBe('hello-world');
  });

Make the test pass. Only implement what the test requires.

Commands, Skills & GitHub CLI

3:30 – 4:30 · 60 minutes

If you type the same multi-step instructions twice, wrap it in a command

Command: /pr

.claude/commands/pr.md
markdown 
# /pr — Create a pull request

Follow these steps in order:

1. Run all tests: `npm test`
2. If tests fail, stop and report the failures
3. Summarize changes in 2-3 bullet points
4. Create PR: `gh pr create --fill`
5. Post the PR URL as your final output

Commands: /review + /catchup

markdown 
# /review — Code review current changes

Review the current diff for quality issues:

1. Run `git diff` to see all staged and unstaged changes
2. Check every changed file for:
   - Security issues (hardcoded secrets, injection vectors)
   - Missing tests for new logic
   - Style violations per CLAUDE.md conventions
3. Rate the changes: Ship / Needs Work / Block
4. List specific findings with file:line references
markdown 
# /catchup — Rebuild context after a break

Quickly rebuild situational awareness:

1. Read CLAUDE.md and all .claude/rules/ files
2. Check `git log --oneline -10` for recent work
3. Scan for open TODO and FIXME comments in src/
4. Summarize the current project state in 3 bullets
5. Suggest what to work on next

Skill: Reviewer

.claude/skills/reviewer/SKILL.md
yaml+md 
---
name: reviewer
trigger: /review
description: Code review with structured output
context: fork
---

# Reviewer Skill

You are a strict code reviewer. For every review:

1. Read the diff with `git diff HEAD~1`
2. Check each changed file for:
   - Security: hardcoded secrets, injection, auth gaps
   - Quality: missing error handling, untested paths
   - Style: naming, structure, conventions from CLAUDE.md
3. Output a verdict: SHIP / NEEDS_WORK / BLOCK
4. List all findings as `file:line — issue description`
5. If BLOCK, explain what must change before shipping

Subagents, CI & Ship

4:30 – 6:00 · 90 minutes

The top layer: orchestrate, integrate, ship

Ship Protocol

Explore — Read relevant code, understand current state
Plan — Outline intended changes before writing code
Implement — Write code and tests in TDD cycles
Commit — Conventional commit, all hooks must pass
PR — Create pull request, verify CI green, request review
PatternUse CaseContext
Explore agentRead-only recon of a codebase areaForked (isolated)
Task agentImplement a bounded taskForked
Review agentReview changes from another agentForked
Parallel worktreesMultiple agents, different branchesFully isolated

Worktree Setup for Parallel Agents

bash 
# Create a worktree for a feature branch
git worktree add ../fhnw-feat-auth -b feat/auth

# Start a Claude session in the worktree
cd ../fhnw-feat-auth
claude --session-name "auth-feature"

# Meanwhile, in the main worktree, start another session
cd ../fhnw-harness
claude --session-name "main-refactor"

# List active worktrees
git worktree list

# Clean up when done
git worktree remove ../fhnw-feat-auth

Encode Workflows

Ship Protocol
